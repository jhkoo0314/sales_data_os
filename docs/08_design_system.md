# 08. Design System

작성일: 2026-03-30  
상태: `draft`

## 1. 문서 목적

이 문서는 웹 앱의 화면 스타일과 공통 UI 규칙을 정리한다.

쉽게 말하면
`화면을 어떤 분위기와 규칙으로 만들 것인가`
를 정하는 문서다.

## 2. 디자인 방향

이 제품은 운영 콘솔이지만,
지루하고 무거운 내부툴처럼 보이면 안 된다.

방향은 아래처럼 잡는다.

- 차분하지만 날카로운 정보 구조
- 숫자와 상태를 빠르게 읽을 수 있는 화면
- 실행과 결과가 명확히 구분되는 레이아웃
- 보고서/실행/아티팩트/에이전트가 서로 다른 성격으로 보이는 구조
- **[New] Cinematic & Editorial Aesthetics (영화적 깊이감과 편집적 명확성)**

## 3. 디자인 키워드

- `Operational`
- `Structured`
- `Clear`
- `Signal-first`
- `Report-aware`
- **`Cinematic` (영화적 대비와 몰입감)**
- **`Editorial` (잡지처럼 읽히는 비즈니스 문장)**
- **`Action-Driven` (사용자의 판단을 돕는 즉각적 액션 포인트)**

## 4. 컬러 방향

## 4.1 기본 배경

- 밝은 중성 계열
- 완전 흰색보다는 약간 부드러운 배경
- **[Cinematic Addon] Dark Mode Midnight Base (#0F172A) & Gradient Mesh (그라데이션 매쉬)**

## 4.2 핵심 포인트

- 청록 또는 딥 블루 계열

이 색은 아래에 주로 쓴다.

- 현재 회사 강조
- 활성 탭
- 주요 액션 버튼
- 선택 상태

## 4.3 상태 색상

- `PASS`: 그린
- `APPROVED`: 청록 또는 블루-그린
- `WARN`: 주황
- `FAIL`: 레드
- `RUNNING`: 블루
- `READY`: 중성 그레이

## 4.4 색상 사용 원칙

- 색만으로 의미를 전달하지 않는다.
- 텍스트와 아이콘을 함께 쓴다.
- 상태 색은 배지/라인/아이콘 정도로 제한한다.

## 5. 타이포그래피 방향

## 5.1 기본 원칙

- 기본은 읽기 쉬운 산세리프
- 숫자와 상태는 더 선명하게
- 제목과 설명의 위계를 크게 둔다

## 5.2 텍스트 위계

- 페이지 제목
- 섹션 제목
- 카드 제목
- 본문 설명
- 보조 메타

이 위계가 분명해야
운영 콘솔이 복잡해 보여도 읽기 쉬워진다.

## 5.3 숫자 표현

- 점수
- 상태 수치
- run 개수
- artifact 개수

이런 숫자는 일반 본문보다 더 선명하게 보이게 한다.

## 6. 레이아웃 원칙

## 6.1 App Shell

기본 구조:

- 좌측 메뉴 (또는 상단 네비게이션)
- 상단 컨텍스트 바
- 메인 콘텐츠 영역 (Canvas)

## 6.2 Cinematic Layout Pattern

- **Top-Nav Structure:** 좌측 사이드바를 제거하고 상단 네비게이션을 사용하여 캔버스의 가로 폭을 극대화한다.
- **Bento Grid:** 정보의 중요도에 따라 비정형적인 크기의 그리드(Bento Box)를 배치하여 위계를 나눈다.
- **Atmospheric Background:** 단순 단색이 아닌, 유리 질감(Glassmorphism)과 미세한 글로우(Glow) 효과를 배경에 둔다.

## 6.2 컨텍스트 고정

항상 보여야 하는 것:

- 현재 회사
- 현재 페이지 제목
- 가능하면 현재 run

## 6.3 화면 순서

각 화면은 아래 순서로 읽히게 한다.

1. 지금 어디를 보고 있는지
2. 현재 상태가 어떤지
3. 어떤 행동을 할 수 있는지
4. 상세 데이터는 무엇인지

## 8. 상태 표현 원칙

### 8.1 배지 일관성

상태 텍스트는 아래를 일관되게 쓴다.

- `READY`
- `RUNNING`
- `PASS`
- `WARN`
- `FAIL`
- `APPROVED`

### 8.2 설명 문장 동반

상태 배지 아래 또는 옆에는
짧은 설명 문장을 같이 둔다.

예:

- `WARN`
- `담당자 배치 불균형으로 운영 점검이 필요합니다.`

### 8.3 단계 상태 시각화

run 상세에서는
단계별 상태가 한눈에 보이도록
수직 리스트나 스텝 카드 형태가 좋다.

### 8.4 Editorial Verdict 시스템

- 시스템 상태를 기계적인 코드가 아닌, 비즈니스 관점의 **'판결문(Verdict)'** 형식으로 제공한다.
- 예: "건강보험공단 데이터와 CRM 데이터의 기간이 일치하지 않습니다. (2025.12 기준 가동 가능)"
- 이 문장은 가장 크고 명확한 영역(보통 Cinematic Panel)에 배치한다.

## 9. 페이지별 디자인 원칙

## 9.1 Workspace

- 큰 요약 카드 중심
- 최근 상태가 가장 먼저 보이게
- 다음 행동 버튼을 상단에 둔다

## 9.2 Upload

- 파일 유형별 섹션 분리
- 저장 상태와 업로드 상태를 구분
- 월별 raw 영역은 일반 업로드와 시각적으로 다르게 구분

## 9.3 Pipeline

- 실행 모드 선택과 현재 상태를 분리
- 실행 버튼은 명확하게 강조
- 진행 중일 때는 현재 단계가 가장 크게 보여야 함

## 9.4 Runs

- 리스트 가독성 우선
- 상태/점수/시간이 빠르게 비교되도록 구성

## 9.5 Reports

- 카드형 목록이 적합
- 생성된 보고서와 미생성 보고서를 한눈에 구분

## 9.6 Agent

- 질문/응답/근거 패널이 명확히 나뉘어야 한다
- 현재 run 정보가 상단에 고정되는 편이 좋다

## 10. 보고서 화면 원칙

- HTML 보고서는 카드형 목록으로 먼저 보여준다.
- 생성되지 않은 보고서는 비활성 카드로 보인다.
- 보고서 열기와 다운로드를 분리한다.
- 어떤 run 기준 보고서인지 함께 보여준다.

## 11. 표와 리스트 원칙

- 너무 많은 열을 기본 화면에 넣지 않는다.
- 중요한 것은 앞쪽 열에 둔다.
- 상태, 시간, 이름, 액션 순으로 정리하는 편이 좋다.

## 12. 모바일 대응 원칙

주 사용 환경은 데스크톱이지만,
최소한 아래는 모바일에서도 깨지지 않아야 한다.

- 회사 선택
- 상태 확인
- run 목록
- 보고서 목록

복잡한 표는 모바일에서 카드형으로 바꿀 수 있다.

## 13. 접근성 원칙

- 키보드 이동 가능
- 대비 충분
- 상태 배지에 텍스트 포함
- 폼 오류는 문장으로 표시
- 버튼은 의미가 분명해야 함

## 14. 디자인 토큰 초안

초기에는 아래 토큰을 정의하는 것이 좋다.

- color
- spacing
- radius
- shadow
- font size
- font weight

## 15. 디자인 시스템 체크리스트

- 현재 회사와 현재 run이 항상 보이는가
- 상태가 색만으로 전달되지 않는가
- 정보 위계가 분명한가
- 운영 콘솔답게 빠르게 읽히는가
- 보고서와 artifact가 구분돼 보이는가
