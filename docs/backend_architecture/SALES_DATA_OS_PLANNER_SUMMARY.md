# Sales Data OS 백엔드 설계 문서 03. 기획자 공유 요약

작성일: 2026-03-31  
문서 구분: 백엔드 설계 문서  
대상: 기획자, 운영 담당자, 비개발 이해관계자 공유용  
작성 기준: 짧고 쉽게 전체 구조를 설명하는 요약 문서

## 1. 이 프로젝트를 한 줄로 말하면

`Sales Data OS`는  
영업 데이터를 받아 정리하고 계산하고 검증한 뒤 최종 보고서까지 연결하는 운영 체계입니다.

즉 단순 웹화면이 아니라,  
`데이터가 들어오면 뒤에서 자동으로 흐르는 운영 시스템`입니다.

---

## 2. 지금 중요한 것은 화면보다 백엔드 로직

현재 웹 화면 뼈대는 먼저 잡혀 있습니다.

지금 더 중요한 것은 아래입니다.

- 어떤 파일을 입력으로 받는가
- 입력이 괜찮은지 누가 먼저 점검하는가
- 회사마다 다른 파일 형식을 어떻게 흡수하는가
- KPI는 어디서 계산하는가
- 계산 결과를 누가 검증하는가
- 어떤 기준으로 다음 단계로 넘기는가
- Builder는 무엇을 받아 보고서를 만드는가

즉 `화면 추가`보다  
`뒤에서 돌아가는 운영 로직 정리`가 우선입니다.

---

## 3. 공식 흐름

현재 공식 흐름은 아래입니다.

`입력 -> 검증 -> 정규화 -> KPI 계산 -> validation -> result asset / payload -> builder`

쉽게 풀면:

1. 파일을 받는다
2. 파일 상태를 먼저 점검한다
3. 회사마다 다른 형식을 공통 형식으로 바꾼다
4. KPI를 계산한다
5. 계산 결과가 다음 단계로 갈 수 있는지 검증한다
6. 표준 결과 자산으로 정리한다
7. Builder가 읽어서 HTML 보고서를 만든다

---

## 4. 앞단 검증과 뒤단 validation은 다르다

이 부분이 가장 중요합니다.

### 앞단 검증

입력 파일이 제대로 들어왔는지 보는 단계입니다.

예:

- 파일이 있는지
- 필수 컬럼이 있는지
- 날짜 형식이 읽히는지
- 기간이 어느 정도 겹치는지
- 자동 수정으로 살릴 수 있는지

즉 `입력 점검`입니다.

### 뒤단 validation

KPI 계산이 끝난 뒤에,  
이 결과를 다음 단계로 넘겨도 되는지 판단하는 단계입니다.

예:

- 품질이 충분한지
- 운영상 경고만 있는지
- 실제 차단해야 하는지
- RADAR나 Builder로 넘겨도 되는지

즉 `전달 판단`입니다.

---

## 5. 9개 모듈을 아주 쉽게 설명하면

### `intake`

입력 파일을 처음 검사하고 정리하는 입구입니다.

### `kpi`

공식 숫자를 계산하는 계산기입니다.

### `crm`

영업 활동 데이터를 행동 KPI 자산으로 만드는 출발 모듈입니다.

### `sandbox`

CRM, 실적, 목표를 묶어 통합 분석 자산을 만드는 모듈입니다.

### `territory`

담당자와 권역 관점으로 실행 구조를 보는 모듈입니다.

### `prescription`

처방/출고 흐름을 검증 자산으로 만드는 모듈입니다.

### `validation`

계산 결과가 다음 단계로 넘어가도 되는지 판단하는 운영 통제 레이어입니다.

### `radar`

중요 신호와 우선순위를 정리하는 인텔리전스 레이어입니다.

### `builder`

계산이 끝난 내용을 화면/HTML 보고서로 바꾸는 마지막 단계입니다.

---

## 6. KPI 계산 원칙

KPI는 여기저기서 다시 계산하면 안 됩니다.

이유는 단순합니다.

- 같은 숫자가 화면마다 다르게 나오면 기준이 흔들립니다.
- 숫자 수정이 필요할 때 한 곳만 바꾸는 구조가 더 안전합니다.

그래서 공식 KPI 계산은 `modules/kpi/*`만 담당합니다.

중요한 뜻:

- Builder는 KPI를 계산하지 않습니다
- Sandbox는 CRM KPI를 다시 계산하지 않습니다
- Territory도 다시 계산하지 않습니다
- RADAR도 KPI를 다시 계산하지 않습니다

---

## 7. Builder는 무엇을 하는가

Builder는 계산기가 아닙니다.

Builder는 이미 앞에서 계산되고 검증된 결과를 받아서  
최종 HTML 보고서로 바꾸는 역할만 합니다.

즉 Builder의 역할은:

- payload 읽기
- 템플릿에 값 넣기
- HTML 생성

아래는 Builder가 하면 안 됩니다.

- raw 직접 해석
- KPI 재계산
- 비즈니스 규칙 재구현

---

## 8. 왜 result asset이 중요한가

`result asset`은  
각 모듈이 만든 결과를 다음 단계로 안전하게 넘기기 위한 표준 상자입니다.

쉽게 말하면:

- 계산 결과를 아무 형식으로 넘기지 않고
- 다음 단계가 읽을 수 있는 공통 산출물로 정리하는 것

대표 예:

- `crm_result_asset`
- `sandbox_result_asset`
- `territory_result_asset`
- `prescription_result_asset`
- `radar_result_asset`

이 구조가 중요한 이유는:

- 모듈 책임이 섞이지 않음
- 다음 단계가 앞단 계산을 다시 하지 않아도 됨
- Builder가 읽을 기준이 분명해짐

---

## 9. 실제 운영에서 사용자가 체감하는 흐름

사용자는 보통 이렇게 보게 됩니다.

1. 회사 선택
2. 파일 업로드
3. intake 결과 확인
4. 필요 시 기간 차이/주의사항 확인
5. 파이프라인 실행
6. 단계별 PASS/WARN/APPROVED 확인
7. 보고서 열기/다운로드

즉 사용자는 화면만 보지만,  
뒤에서는 아래가 순서대로 일어납니다.

- 입력 점검
- 자동 보정
- 표준화
- KPI 계산
- validation
- result asset 생성
- payload 생성
- Builder 렌더링

## 10. 기획 관점에서 꼭 맞춰야 하는 메시지

이 프로젝트를 설명할 때는 아래처럼 말하는 것이 맞습니다.

### 맞는 설명

- `Sales Data OS`
- `Validation Layer`
- `Builder Preview`
- `Validation Result`

### 피해야 하는 설명

- validation이 전체 시스템이다
- validation이 KPI를 계산한다
- Builder가 KPI를 다시 계산한다
- RADAR가 KPI를 다시 계산한다

## 11. 현재 우선순위 제안

지금 기획/설계 우선순위는 아래 순서가 맞습니다.

1. 백엔드 운영 로직 명세 고정
2. API 흐름 정의
3. 상태값 정의
4. 화면과 API 연결
5. 이후 화면 상세 개선

즉 지금은 화면을 더 예쁘게 만드는 단계보다,  
`무엇을 입력받고 어떤 판단을 거쳐 무엇을 보여줄지`를 먼저 고정하는 단계입니다.

## 12. 최종 한 줄 정리

`Sales Data OS Web`은 단순 웹앱이 아니라,  
`영업 데이터를 받아 정리하고 계산하고 검증한 뒤 최종 보고서까지 연결하는 운영 체계의 웹 입구`로 이해하는 것이 맞습니다.
