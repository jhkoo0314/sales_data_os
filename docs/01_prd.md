# 01. PRD

작성일: 2026-03-30  
상태: `draft`

## 1. 문서 목적

이 문서는 현재 루트에서 시작할
`Sales Data OS Web` 프로젝트의 제품 요구사항 문서다.

쉽게 말하면 아래를 먼저 확정하는 문서다.

- 왜 이 웹 제품이 필요한가
- 누가 쓰는가
- 사용자는 무엇을 할 수 있어야 하는가
- 무엇을 먼저 만들고 무엇은 나중으로 미루는가
- 이 웹 제품이 절대 하면 안 되는 것은 무엇인가

이 문서는 구현보다 먼저 읽는 기준 문서다.

## 2. 제품 한 줄 정의

`Sales Data OS Web`은
기존 Python 기반 Sales Data OS 계산 엔진과 Validation Layer를 유지하면서,
운영 입력, 실행 통제, 실행 상태 확인, 결과 자산 탐색, 보고서 열람, Agent 사용 경험을
브라우저 기반 웹 앱으로 제공하는 운영 제품이다.

## 3. 배경

현재 `Sales Data OS`는 Streamlit 운영 콘솔을 통해
원천데이터 업로드, 파이프라인 실행, 검증 결과 확인, HTML 보고서 열람까지 가능하다.

하지만 이 방식은 아래 한계가 있다.

### 3.1 제품 수준 화면 설계 한계

- 화면 이동 구조가 제한적이다.
- 페이지 단위 설계보다 한 화면에 많은 요소가 모이기 쉽다.
- 운영 콘솔을 장기적으로 확장할수록 구조가 무거워진다.

### 3.2 상태 관리와 실행 흐름 한계

- 업로드 상태
- 저장 상태
- 실행 상태
- 과거 run 이력

이 네 가지를 명확히 나누어 보여주기 어렵다.

### 3.3 확장성 한계

앞으로 아래가 붙을 가능성이 높다.

- 인증
- 권한
- 실행 이력 비교
- 협업용 접근 제어
- 운영자 중심의 정식 웹 앱 UX

이런 확장은 Streamlit보다 웹 프론트엔드에서 더 자연스럽다.

### 3.4 역할 분리 필요

현재 시스템에서 중요한 것은
`계산은 Python`,
`표현은 웹`
구조를 더 선명하게 만드는 것이다.

즉 이번 작업은
계산 엔진을 뜯어고치는 작업이 아니라
운영 제품의 겉면과 사용자 경험을 정식 웹 구조로 바꾸는 작업이다.

## 4. 문제 정의

현재 사용자는 아래 문제를 겪는다.

### 문제 1. 현재 상태를 빠르게 이해하기 어렵다

사용자는 지금 보고 있는 회사, 실행 상태, 마지막 run, 생성된 보고서를
더 명확하게 보고 싶어 한다.

### 문제 2. 업로드와 실행 흐름이 제품적으로 분리되어 있지 않다

파일을 올리고,
저장하고,
실행하고,
결과를 확인하는 흐름을
더 단계적으로 이해하기 쉬운 화면으로 나눌 필요가 있다.

### 문제 3. 장기 실행 작업 UX가 약하다

파이프라인 실행은 짧지 않을 수 있다.
웹에서는 지금 어느 단계인지,
계속 기다리면 되는지,
어디서 막혔는지를 더 분명히 보여줘야 한다.

### 문제 4. 결과 탐색 경험이 제한적이다

사용자는 아래를 쉽게 찾고 싶다.

- 최신 run
- 과거 run
- 단계별 상태
- artifact
- 보고서
- Agent 근거 자료

### 문제 5. 장기적으로 제품형 웹 앱으로 가기 어렵다

운영 도구가 아니라
“실제로 계속 쓸 웹 제품”으로 가려면
웹 프론트엔드 기반 구조가 필요하다.

## 5. 제품 목표

이 프로젝트의 목표는 아래 다섯 가지다.

### 목표 1. Python 계산 엔진 유지

KPI 계산, validation, builder 산출은 계속 Python이 맡는다.

### 목표 2. 웹 운영 콘솔 구축

운영자가 브라우저에서 아래를 수행할 수 있어야 한다.

- 회사 선택
- raw 업로드
- 실행 요청
- 실행 상태 확인
- 결과 조회

### 목표 3. run 중심 UX 정리

웹은 “현재 화면”보다 “어떤 실행(run)을 보고 있는지”가 더 분명해야 한다.

### 목표 4. 결과 탐색성과 설명력 강화

단순 상태 배지만 보여주는 것이 아니라
왜 PASS/WARN/FAIL인지 사람이 읽을 수 있는 설명을 함께 제공해야 한다.

### 목표 5. 장기적 웹 제품 기반 확보

나중에 인증, 권한, 조직별 접근 제어, 더 나은 보고서 UX를 붙일 수 있는 구조를 만든다.

## 6. 제품 목표가 아닌 것

이번 프로젝트에서 하지 않는 것은 아래다.

### 6.1 KPI 계산 엔진 재작성

- JavaScript로 KPI 계산 이전
- 프론트엔드에서 공식 계산식 복제

이것은 하지 않는다.

### 6.2 Builder 역할 변경

- Builder에서 KPI 재계산
- Builder를 분석 엔진처럼 확장

이것은 하지 않는다.

### 6.3 validation 용어 기준

- `OPS`라는 표현은 더 이상 중심 용어로 사용하지 않음
- 검증과 전달 판단 레이어는 `validation`으로만 설명

이것은 하지 않는다.

### 6.4 한 번에 전체 치환

처음부터 Streamlit을 완전히 제거하지 않는다.
초기에는 병행 검증을 허용한다.

### 6.5 대규모 백엔드 재구성

초기 단계에서는 Python 내부 구조를 전면 개편하지 않는다.
웹이 필요한 API를 안정적으로 연결하는 것이 우선이다.

## 7. 대상 사용자

이 제품은 아래 세 그룹을 중심으로 설계한다.

### 7.1 운영 관리자

이 사용자는 실제 운영 흐름을 관리한다.

주요 행동:

- 회사 선택
- 원천 파일 업로드
- 저장 상태 확인
- 실행 모드 선택
- 파이프라인 실행
- 실행 결과 확인

이 사용자가 중요하게 보는 것:

- 지금 실행 가능한가
- 어떤 파일이 부족한가
- 지금 어디까지 실행됐는가
- 어떤 단계에서 막혔는가

### 7.2 분석 사용자

이 사용자는 실행 결과를 해석한다.

주요 행동:

- 단계별 결과 확인
- PASS/WARN/FAIL 이유 확인
- artifact 탐색
- 보고서 확인
- run 비교

이 사용자가 중요하게 보는 것:

- 왜 이 판정이 나왔는가
- 어떤 숫자가 근거인가
- 어떤 보고서를 바로 열 수 있는가

### 7.3 관리자/리더

이 사용자는 전체 상태를 빠르게 확인한다.

주요 행동:

- 최근 실행 요약 확인
- 경고 상태 확인
- 승인된 결과 확인
- 최종 보고서 확인

이 사용자가 중요하게 보는 것:

- 지금 상태가 정상인가
- 운영상 문제가 있는가
- 결과물이 준비됐는가

## 8. 사용자 시나리오

### 시나리오 A. 운영자가 월별 raw를 업로드하고 통합 실행한다

1. 운영자가 회사 선택
2. Upload 화면에서 월별 raw 업로드
3. 저장된 파일 목록 확인
4. Pipeline 화면에서 `통합 실행` 선택
5. 실행 시작
6. 단계별 진행 상태 확인
7. 완료 후 Reports 화면에서 보고서 확인

### 시나리오 B. 분석 사용자가 최신 run의 WARN 이유를 확인한다

1. 회사 선택
2. Runs 화면에서 최신 run 선택
3. run 상세에서 단계별 상태 확인
4. WARN 단계 클릭
5. 이유 문장과 근거 수치 확인
6. 필요한 artifact 열람

### 시나리오 C. 관리자가 보고서만 빠르게 확인한다

1. 회사 선택
2. Workspace 또는 Reports 진입
3. 최근 생성 보고서 목록 확인
4. 원하는 보고서 열기

### 시나리오 D. 사용자가 Agent에 질문한다

1. 회사 선택
2. 특정 run 선택
3. Agent 화면에서 질문 입력
4. 응답 확인
5. 근거 artifact 확인

## 9. 핵심 사용자 질문

이 제품은 아래 질문에 빠르게 답해야 한다.

- 지금 어느 회사를 보고 있는가
- 어떤 파일이 준비되어 있는가
- 지금 실행 가능한가
- 실행 중인가, 끝났는가
- 어디 단계에서 경고가 났는가
- 왜 그런 판정이 나왔는가
- 어떤 결과 파일과 보고서를 볼 수 있는가
- 어떤 run을 기준으로 말하고 있는가

## 10. 제품 원칙

### 10.1 시스템 이름 원칙

시스템 전체 이름은 항상 `Sales Data OS`다.

### 10.2 validation 역할 원칙

`validation`은 시스템 전체 이름이 아니라
KPI 결과를 검증하고 다음 단계 전달 가능 여부를 판단하는 레이어다.

### 10.3 KPI 단일 소스 원칙

KPI 계산은 계속 `modules/kpi/*`를 단일 소스로 유지한다.

### 10.4 Builder 원칙

Builder는 render-only다.
웹도 Builder를 계산기로 취급하면 안 된다.

### 10.5 company_key 원칙

모든 실행과 저장 기준은 `company_key`다.

### 10.6 run 중심 원칙

웹에서는 “현재 페이지”보다
“현재 어떤 run을 보고 있는가”를 더 중요하게 다룬다.

## 11. 제품 범위

### 11.1 MVP 범위

아래는 1차 웹 버전에 반드시 포함한다.

- 회사 선택
- raw 업로드
- 저장된 입력 상태 확인
- 실행 모드 선택
- 파이프라인 실행 시작
- run 상태 조회
- 단계별 결과 요약
- 최신 보고서 목록 확인
- 보고서 열람

### 11.2 2차 범위

- runs 목록
- run 상세 비교
- artifact 상세 탐색
- Agent 화면
- 더 나은 상태 필터

### 11.3 3차 범위

- 인증
- 권한
- 조직별 접근 제어
- 더 고급 대시보드
- 운영 알림

## 12. 상세 기능 요구사항

### 12.1 회사 선택

사용자는 현재 작업 대상을 분명히 선택할 수 있어야 한다.

필수 요구사항:

- 현재 선택 회사 표시
- 회사 목록 조회
- 회사 검색 가능
- 회사 변경 시 관련 화면 갱신

### 12.2 업로드

사용자는 raw 파일을 올리고 저장 상태를 확인할 수 있어야 한다.

필수 요구사항:

- 파일 유형별 업로드
- 월별 raw 다중 업로드
- 저장 전/후 상태 구분
- 업로드 후 저장 목록 확인
- 실행 전에 부족 파일 확인

### 12.3 실행

사용자는 실행 모드를 선택하고 파이프라인을 시작할 수 있어야 한다.

필수 요구사항:

- 실행 모드 선택
- 실행 시작
- `run_id` 생성
- 단계별 진행 상태 표시
- 완료/경고/실패 결과 표시

### 12.4 결과 요약

사용자는 실행 결과를 한눈에 볼 수 있어야 한다.

필수 요구사항:

- 전체 상태
- 전체 점수
- 단계별 상태
- 설명 문장
- 근거 수치 요약

### 12.5 보고서

사용자는 생성된 보고서를 쉽게 찾고 열 수 있어야 한다.

필수 요구사항:

- 보고서 목록
- 생성 여부 표시
- 열기
- 다운로드

### 12.6 Agent

2차 범위지만 요구사항은 미리 고려한다.

필수 요구사항:

- run 선택
- 질문 입력
- 응답 확인
- 근거 artifact 표시
- 대화 이력 표시

## 13. 비기능 요구사항

### 13.1 명확성

상태는 배지 하나로 끝내지 않는다.
반드시 설명 문장을 함께 보여준다.

### 13.2 안정성

긴 실행 작업이라도 사용자가 화면에서 길을 잃지 않아야 한다.

### 13.3 확장성

추후 인증, 권한, 고급 비교 기능을 붙일 수 있어야 한다.

### 13.4 책임 분리

프론트엔드는 계산을 하지 않는다.
Python은 화면 책임을 갖지 않는다.

### 13.5 추적 가능성

모든 결과는 company_key + run_id 기준으로 추적 가능해야 한다.

## 14. 성공 지표

### 운영 성공 지표

- 운영자가 웹에서 업로드부터 실행까지 완료할 수 있다.
- 운영자가 웹에서 현재 상태를 혼동하지 않는다.

### 제품 성공 지표

- 핵심 운영 흐름에서 Streamlit 없이도 1차 사용 가능하다.
- run 상태와 보고서 연결이 웹에서 자연스럽다.

### 구조 성공 지표

- KPI 계산 로직이 프론트에 복제되지 않았다.
- Builder render-only 원칙이 유지된다.

## 15. 리스크와 대응

### 리스크 1. 웹에서 실행 대기 경험이 답답할 수 있다

대응:

- polling 기반 상태 조회
- 단계명 표시
- 마지막 갱신 시각 표시

### 리스크 2. 업로드와 저장 개념이 혼동될 수 있다

대응:

- 임시 업로드
- 저장 완료
- 실행 반영

이 세 상태를 분리해서 보여준다.

### 리스크 3. 보고서 표시 방식이 애매할 수 있다

대응:

- 초기에는 기존 HTML 보고서를 우선 연결
- 이후 필요 시 React 재구성 판단

### 리스크 4. 프론트와 백엔드 역할이 섞일 수 있다

대응:

- API 책임 문서화
- 프론트 계산 금지 원칙 고정

## 17. 최종 정의

`Sales Data OS Web`은
기존 Python 기반 Sales Data OS 계산/검증 구조를 유지하면서,
운영 콘솔 경험을 제품 수준의 웹 앱으로 전환하는 프로젝트다.
