# 추후 보완사항: 약국-병원 링크(문전) + 담당자 배정 모델

작성일: `2026-03-31`  
문서 목적: 현재 포트폴리오/데모 구현에서 `prescription`이 유난히 원본과 맞추기 어려웠던 이유와, 실제 운영 적용 시 “안전하고 유지보수 가능한 구조”를 남긴다.

## 1. 문제 요약(왜 처방이 특히 어려웠나)

`prescription`은 입력 단위가 “약국”이고, 다른 모듈(CRM/Territory/Target)은 주로 “병원(거래처)” 단위로 운영되는 경우가 많다.  
즉, 처방 데이터가 들어왔을 때 “누구(담당자)에게 귀속할지”를 결정하려면 약국 → 병원(문전) → 담당자 연결이 필요하다.

추가로, CSV에서 주소처럼 셀 안에 `,`가 들어갈 때 파싱이 깨지면 금액/수량 컬럼이 조용히 틀어지고, 그 결과가 월별 분포(그래프)까지 크게 흔들린다.

## 2. 핵심 운영 시나리오(사용자가 말한 케이스)

“A 담당자가 A 병원에 문전약국 C, D 실적을 주고 싶다.”

이걸 시스템이 안정적으로 하려면, 아래가 전제되어야 한다.

- 약국마다 고유한 `pharmacy_id`가 존재한다
- 약국이 어떤 병원에 문전인지 `pharmacy_id -> hospital_id`로 연결되어 있다
- 병원 담당자는 `hospital_id -> rep_id`로 배정되어 있다

그러면 담당자가 바뀌었을 때는 “병원 배정표만 갱신”하면 처방 귀속도 함께 바뀐다.

## 3. 단순 결론(하지만 1가지를 더 넣어야 안전)

`pharmacy_id`와 `hospital_id`를 묶어두고, 담당자 변경 시 `rep_id`만 바꾸면 대부분 해결된다.

단, 현실 운영에서는 담당자 변경이 “언제부터 적용되는지”가 중요하므로 **기간(적용 시작/종료)** 정보가 반드시 필요하다.

기간이 없으면, 과거 데이터까지 새 담당자에게 붙는 문제가 생긴다.

## 4. 권장 데이터 구조(추후 실제 적용용)

아래 3개가 최소 세트다.

1. `pharmacy_master`
- 약국 고유 키: `pharmacy_id` (내부 키)
- 표시명: `pharmacy_name`
- 정규화 키: `pharmacy_name_key` (띄어쓰기/오타 흔들림 흡수)
- 보조정보: 주소/위경도/지역(선택)

2. `pharmacy_to_hospital_link` (문전 링크)
- `pharmacy_id`
- `hospital_id`
- `link_type` (예: `front_pharmacy`, `nearby`, `unknown`)
- `effective_start_month`, `effective_end_month`
- `confidence` (수동/자동, 신뢰도)

3. `crm_account_assignment` (병원 담당 배정)
- `hospital_id`
- `rep_id`, `branch_id`, `territory_*`
- `effective_start_month`, `effective_end_month` (권장)

## 5. 파이프라인에 넣는 위치(변경 지점)

현재 시스템 흐름을 유지하면서, “귀속 결정”만 바꾸는 것이 안전하다.

- 변경해야 하는 곳: `prescription`에서 `rep_id/territory`를 붙이는 단계
  - 기존: 지역/룩업/보정 규칙 기반 fallback
  - 추후: `pharmacy_id -> hospital_id -> rep_id` 경로를 우선 사용

- 바꾸지 말아야 하는 곳:
  - Builder 템플릿 디자인(렌더 전용)
  - payload 계약(템플릿이 기대하는 필드 구조)
  - validation의 역할(판정/설명 레이어)

## 6. 포트폴리오(원본 동기화)와 실제 운영 적용의 차이

현재는 포트폴리오 목적이라 원본 프로젝트(`sfe_master_ops`)와 수치/형태를 맞추는 것이 중요했다.

하지만 실제 운영에서 “문전 약국 귀속”을 도입하면 아래는 정상이다.

- 원본과 숫자가 달라진다(귀속 규칙이 달라지므로)
- 대신 사용자 입장에서 원하는 “누가 이 실적을 가져가야 하는가”를 더 정확히 설명할 수 있다

## 7. 다음에 할 일(구체)

- `pharmacy_master` 생성 규칙 결정(이름+주소 기반, 재실행해도 같은 ID가 나오게)
- 문전 링크 테이블 수집/입력 방식 결정(수동 입력 UI 또는 엑셀 업로드)
- `effective_*` 기간 도입(담당자 변경/문전 변경 반영)
- prescription 귀속 룰을 “병원 기준”과 “약국 기준” 중 선택 가능하게 분기(운영 정책에 따라)
