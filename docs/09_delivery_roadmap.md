# 09. Delivery Roadmap

작성일: 2026-03-30  
상태: `draft`

## 1. 문서 목적

이 문서는 새 웹 프로젝트를 어떤 순서로 구현할지 정리한다.

쉽게 말하면
`무엇을 먼저 만들고, 무엇은 나중에 만들지`
를 정하는 문서다.

## 2. 개발 전략

이 프로젝트는 한 번에 크게 만들면 위험하다.

그래서 아래 전략으로 간다.

### 2.1 문서 고정 후 구현

먼저 문서로 범위를 고정하고,
그다음 실제 코드를 만든다.

### 2.2 운영 핵심 흐름부터 구현

가장 중요한 것은 아래다.

- 회사 선택
- 업로드
- 실행
- run 상태 확인
- 보고서 확인

### 2.3 점진 전환

초기에는 Streamlit과 병행 검증을 허용한다.
웹이 안정화된 뒤 점차 중심을 옮긴다.

## 3. Phase 0. 문서와 범위 고정

### 목표

현재 루트에서 구현을 시작하기 전에
기준 문서를 고정한다.

### 작업 항목

- PRD 확정
- User Flow 확정
- IA 확정
- 기술 스택 확정
- 프론트 구조 확정
- API 범위 확정
- 데이터 흐름 확정

### 산출물

- 상세 문서 세트

### 완료 기준

- 개발자가 현재 루트에서 바로 시작할 수 있다.
- 주요 의사결정이 문서에 반영돼 있다.

## 4. Phase 1. 웹 앱 뼈대

### 목표

새 웹 프로젝트의 기본 틀을 만든다.

### 작업 항목

- Next.js 프로젝트 생성
- `pnpm` 기반 루트 시작 세팅
- TypeScript 설정
- Tailwind 설정
- 기본 App Shell 생성
- 라우트 뼈대 생성
- 공통 상태 구조 생성

### 산출물

- 초기 웹 프로젝트
- Sidebar
- Header
- 회사 문맥 표시 바

### 완료 기준

- 빈 화면이 아니라 제품 형태의 기본 구조가 있다.
- 회사 선택 문맥을 담을 수 있다.

## 5. Phase 2. 회사 선택과 업로드

### 목표

입력 흐름을 웹에서 먼저 성립시킨다.

### 작업 항목

- 회사 목록 조회
- 회사 선택 기능
- Upload 화면
- 일반 파일 업로드
- 월별 raw 업로드
- 업로드 목록 조회

### 산출물

- 업로드 가능한 웹 화면
- 저장된 입력 상태 화면

### 완료 기준

- 운영자가 웹에서 파일을 올릴 수 있다.
- 어떤 파일이 저장됐는지 확인할 수 있다.

## 6. Phase 3. 실행 시작과 run 상태 추적

### 목표

웹에서 파이프라인 실행과 상태 확인이 가능해야 한다.

### 작업 항목

- 실행 모드 선택 UI
- 실행 요청 등록 연결
- `run_id` 생성 흐름 연결
- polling 상태 조회
- 실행 중 화면
- 실행 완료 화면

### 산출물

- Pipeline 화면
- run 상태 요약 카드

### 완료 기준

- 운영자가 웹에서 실행을 시작할 수 있다.
- 실행 중/완료 상태를 웹에서 추적할 수 있다.

## 7. Phase 4. Runs와 결과 탐색

### 목표

실행 결과를 run 중심으로 탐색할 수 있어야 한다.

### 작업 항목

- run 목록 화면
- run 상세 화면
- 단계별 상태 표시
- reasoning 문장 표시
- artifact 목록 연결

### 산출물

- Runs 목록
- Run Detail 화면
- Artifact 목록

### 완료 기준

- 과거 run과 현재 run을 구분해 볼 수 있다.
- WARN/FAIL 이유를 run 상세에서 확인할 수 있다.

## 8. Phase 5. Reports 연결

### 목표

생성된 HTML 보고서를 웹에서 쉽게 열 수 있어야 한다.

### 작업 항목

- 보고서 목록 조회
- 보고서 카드 UI
- 보고서 열기
- 다운로드 연결
- run 기준 연결

### 산출물

- Reports 화면
- 보고서 카드
- 열기/다운로드 액션

### 완료 기준

- 사용자가 웹에서 생성된 보고서를 바로 찾고 열 수 있다.

## 9. Phase 6. Agent 연결

### 목표

run 기반 질문/응답 경험을 웹에 붙인다.

### 작업 항목

- Agent 화면
- run 선택
- 질문 입력
- 응답 표시
- 근거 artifact 패널
- 대화 이력 표시

### 산출물

- Agent MVP

### 완료 기준

- 사용자가 특정 run 기준으로 질문하고 답변을 받을 수 있다.

## 10. Phase 7. 운영 안정화

### 목표

서비스 품질을 높인다.

### 작업 항목

- 에러 처리 고도화
- 로딩 상태 정리
- 테스트 자동화
- 로깅/모니터링
- 필요 시 인증/권한 초안

### 산출물

- 안정화된 운영 버전

### 완료 기준

- 핵심 흐름이 깨지지 않는다.
- 기본 테스트가 준비된다.

## 11. 첫 실제 개발 범위

가장 먼저 만들 범위는 아래로 제한하는 것이 좋다.

1. 회사 선택
2. Upload 화면
3. Pipeline 화면
4. run 상태 확인
5. Reports 목록

이 다섯 가지가 먼저 되면
웹 전환의 핵심 골격이 생긴다.

## 12. 왜 이 순서인가

- 업로드 없이 실행은 의미가 없다.
- 실행 없이 runs는 의미가 없다.
- run 없이 reports는 문맥이 약하다.
- Agent는 기본 결과 탐색이 된 뒤 붙는 것이 맞다.

## 13. 팀 역할 제안

새 프로젝트를 시작할 때는 아래처럼 역할을 나누면 좋다.

### 프론트

- App Shell
- 페이지 구현
- 상태관리
- API 연동

### 백엔드

- Supabase 테이블/스토리지
- Python polling worker
- run 상태 반환 구조
- report/artifact 메타 반환

### 공통

- API 계약
- run 모델
- 상태 명칭 통일

## 14. 리스크 관리 포인트

### 리스크 1. API 정의 없이 화면부터 만들기

대응:

- 최소 응답 구조 먼저 고정

### 리스크 2. 보고서 표시 방식을 너무 늦게 결정

대응:

- 초기에는 HTML 그대로 연결하는 방향 우선

### 리스크 3. run 상태 polling 설계를 미루기

대응:

- Phase 3 전에 응답 구조 먼저 고정

## 15. 단계별 산출물 표

### Phase 0

- 문서 세트

### Phase 1

- 프로젝트 뼈대

### Phase 2

- 업로드 화면

### Phase 3

- 실행과 상태조회 화면

### Phase 4

- runs/artifacts 화면

### Phase 5

- reports 화면

### Phase 6

- agent 화면

### Phase 7

- 안정화와 테스트

## 16. 최종 완료 기준

- 웹에서 업로드부터 실행까지 가능하다.
- run 중심 상태 추적이 된다.
- 보고서 열람이 된다.
- Python 계산 로직을 프론트로 복제하지 않았다.
- Builder render-only 원칙이 유지된다.
