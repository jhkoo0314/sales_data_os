# 02. User Flow

작성일: 2026-03-30  
상태: `draft`

## 1. 문서 목적

이 문서는 사용자가 웹에서 어떤 순서로 행동하는지,
그리고 각 단계에서 무엇을 보고 무엇을 눌러야 하는지 정리한다.

쉽게 말하면
`사용 시나리오와 화면 이동 순서` 문서다.

이 문서는 아래 문서의 기준이 된다.

- 페이지별 기능 명세
- API 명세
- 상태 관리 설계
- UX 설계

## 2. 가장 중요한 핵심 흐름

이 제품의 가장 중요한 기본 흐름은 아래다.

1. 사용자가 회사를 선택한다.
2. 사용자가 raw 파일을 업로드한다.
3. 저장된 입력 상태를 확인한다.
4. 실행 모드를 선택한다.
5. 파이프라인 실행을 시작한다.
6. 현재 run 상태를 확인한다.
7. 완료된 결과를 확인한다.
8. 보고서 또는 artifact를 연다.
9. 필요하면 Agent에 질문한다.

이 흐름이 끊기면 제품 가치가 크게 떨어진다.

## 3. 사용자 유형별 핵심 흐름

### 3.1 운영 관리자 핵심 흐름

운영 관리자는 아래 흐름을 가장 많이 사용한다.

1. 회사 선택
2. Upload 화면 진입
3. 파일 업로드
4. 저장 상태 확인
5. Pipeline 화면 진입
6. 실행 모드 선택
7. 실행 시작
8. run 상태 확인
9. 결과 상태 확인
10. 보고서 생성 여부 확인

운영 관리자에게 가장 중요한 것은 아래다.

- 지금 실행 가능한가
- 파일이 충분한가
- 지금 어느 단계까지 갔는가
- 실패했다면 어디서 실패했는가

### 3.2 분석 사용자 핵심 흐름

분석 사용자는 아래 흐름을 많이 사용한다.

1. 회사 선택
2. Runs 화면 진입
3. 최신 run 선택
4. run 상세 확인
5. WARN/FAIL 단계 선택
6. 이유 문장 확인
7. 근거 수치 확인
8. artifact 또는 보고서 확인

분석 사용자에게 가장 중요한 것은 아래다.

- 왜 이 판정이 나왔는가
- 이 판단의 근거는 무엇인가
- 어떤 결과 파일을 보면 되는가

### 3.3 관리자/리더 핵심 흐름

관리자/리더는 아래 흐름을 자주 쓴다.

1. 회사 선택
2. Workspace 또는 Reports 화면 진입
3. 최근 실행 상태 확인
4. 주요 경고 확인
5. 최종 보고서 열람

이 사용자에게 중요한 것은
세부 처리보다 `요약과 상태`다.

## 4. 상세 기본 흐름

아래는 가장 중요한 end-to-end 흐름이다.

### 4.1 회사 선택

1. 사용자가 웹 앱 진입
2. 현재 선택 회사가 없으면 회사 선택 유도
3. 회사 목록 표시
4. 검색 또는 선택
5. 선택 완료 후 해당 회사 기준 화면 이동

화면에서 보여줘야 할 것:

- 회사명
- company_key
- 최근 실행 상태 요약
- 최근 업로드 상태 요약

### 4.2 업로드

1. 사용자가 Upload 화면으로 이동
2. 업로드 가능한 파일 유형 확인
3. 파일 선택
4. 파일 업로드
5. 임시 상태 또는 저장 예정 상태 확인
6. 저장 완료
7. 업로드 결과 목록 확인

화면에서 보여줘야 할 것:

- 파일 유형
- 파일명
- 업로드 시각
- 저장 상태
- 월별 raw 여부
- source type

### 4.3 실행 준비 확인

1. 사용자가 Pipeline 화면 이동
2. 현재 저장된 파일 목록 확인
3. 실행 가능 여부 확인
4. 부족한 입력이 있으면 경고 표시
5. 실행 모드 선택
6. 실행 전 확인 메시지 표시

화면에서 보여줘야 할 것:

- 현재 회사
- 저장된 파일 목록
- 실행 가능 여부
- 경고/주의 메시지
- 실행 모드 설명

### 4.4 실행 시작

1. 사용자가 실행 버튼 클릭
2. 시스템이 새 run 생성
3. `run_id` 반환
4. 실행 상태 화면으로 전환

보여줘야 할 것:

- run_id
- 실행 시작 시각
- 선택한 실행 모드
- 현재 상태 `Running`

### 4.5 실행 진행 확인

1. 프론트가 polling으로 상태 조회
2. 단계별 진행 상태 갱신
3. 완료 단계는 완료 표시
4. 경고/실패 단계는 강조
5. 전체 상태 갱신

보여줘야 할 것:

- 현재 단계명
- 완료한 단계 수
- 마지막 갱신 시각
- 단계별 상태 배지
- 사람이 읽는 짧은 설명

### 4.6 실행 완료 후 확인

1. 실행 완료
2. 전체 상태 표시
3. 전체 점수 표시
4. 단계별 결과 카드 표시
5. 생성 artifact 개수 요약
6. 생성 보고서 개수 요약

보여줘야 할 것:

- `Pass`, `Warn`, `Fail`, `Approved`
- overall score
- 단계별 reasoning
- 다음 행동 버튼

예:

- `보고서 보기`
- `artifact 보기`
- `Agent로 질문하기`

## 5. 업로드 상세 흐름

## 5.1 일반 raw 업로드

1. 업로드 화면 진입
2. 파일 유형별 업로드 영역 확인
3. 파일 선택
4. 업로드 처리
5. 업로드 성공/실패 표시
6. 저장 목록 갱신

대상 예시:

- CRM 활동
- 담당자/조직 마스터
- 거래처/배정 파일
- 실적
- 목표
- Prescription

## 5.2 월별 raw 업로드

1. 사용자가 월별 업로드 모드 선택
2. 여러 파일 선택
3. 파일명에서 월 정보 파악
4. 업로드 처리
5. 월별 저장 상태 표시

보여줘야 할 것:

- 어떤 월 파일이 들어왔는지
- 중복 업로드인지
- 월 정보 파악 실패 여부

## 5.3 업로드 후 상태 분기

업로드 후에는 아래 상태 중 하나를 보여줘야 한다.

- `uploaded`
- `saved`
- `needs_review`
- `invalid`

이 상태를 구분하는 이유는
“파일은 올렸지만 실행 준비는 아직 아닌 상태”를
분명히 하기 위해서다.

## 6. 실행 상세 흐름

## 6.1 실행 모드 선택 흐름

사용자는 실행 모드를 고를 때
이 모드가 무엇을 실행하는지 이해할 수 있어야 한다.

예시:

- `CRM -> Sandbox`
- `CRM -> Territory`
- `Sandbox -> HTML`
- `CRM -> PDF`
- `통합 실행`

각 모드 옆에 보여줘야 할 것:

- 간단한 설명
- 생성 예상 결과
- 필요한 입력 파일 요약

## 6.2 실행 중 흐름

실행 중에는 아래가 중요하다.

- 지금 돌아가고 있다는 확신
- 멈춘 것이 아니라는 확신
- 어디 단계인지 이해 가능

필수 표시:

- 상태: `Running`
- 현재 단계
- 최근 로그 요약
- 마지막 갱신 시각

## 6.3 실행 완료 흐름

실행이 끝나면 사용자는 아래를 바로 알 수 있어야 한다.

- 성공했는가
- 경고가 있는가
- 실패했는가
- 다음으로 무엇을 볼 수 있는가

필수 액션:

- run 상세 보기
- 보고서 보기
- artifact 보기

## 7. 결과 확인 흐름

## 7.1 run 상세 확인

1. 사용자가 run 선택
2. 전체 상태 확인
3. 단계별 결과 확인
4. 각 단계의 설명 확인
5. 근거 데이터 진입

필수 정보:

- run_id
- company_key
- execution_mode
- started_at
- finished_at
- status
- overall_score

## 7.2 WARN/FAIL 확인 흐름

1. 사용자가 WARN 또는 FAIL 단계 선택
2. 상세 설명 열기
3. 근거 수치 확인
4. 관련 artifact 링크 확인

필수 UX:

- 짧은 기술 코드보다 사람이 읽는 설명 우선
- 근거 수치는 표 또는 요약 카드로 표시

## 8. 보고서 열람 흐름

## 8.1 보고서 목록 흐름

1. Reports 화면 진입
2. 생성 가능한 보고서 카드 표시
3. 각 카드에 생성 상태 표시
4. 생성된 보고서는 열기 가능
5. 미생성 보고서는 비활성 표시

예시 보고서:

- CRM
- Sandbox
- Territory
- Prescription
- RADAR
- Total Validation

## 8.2 보고서 열기 흐름

1. 보고서 카드 선택
2. 보고서 viewer 또는 새 창 열기
3. 다운로드 가능 시 다운로드 제공

필수 정보:

- 보고서명
- 생성 시각
- 어떤 run 기준인지

## 9. Agent 흐름

## 9.1 Agent 진입 흐름

1. Agent 화면 진입
2. 현재 회사 표시
3. 현재 선택 run 표시
4. 질문 입력
5. 응답 생성
6. 근거 artifact 연결

## 9.2 Agent 사용 원칙

Agent는 아무 run이나 기준 없이 답하면 안 된다.

반드시 아래 기준이 보여야 한다.

- 현재 company_key
- 현재 run_id
- 사용한 근거 artifact

## 10. 예외 흐름

## 10.1 회사 미선택

문제:

- 사용자가 아직 회사를 고르지 않음

처리:

- 모든 작업 버튼 비활성
- 회사 선택 유도 화면 표시

## 10.2 입력 부족

문제:

- 실행에 필요한 파일이 부족함

처리:

- 실행 버튼 비활성 또는 경고
- 부족한 파일 목록 표시
- Upload 화면으로 바로 가기 제공

## 10.3 실행 실패

문제:

- 파이프라인 중간 실패

처리:

- 실패 단계 표시
- 짧은 실패 설명 표시
- run 상세로 이동 유도

## 10.4 보고서 미생성

문제:

- run은 끝났지만 특정 보고서가 없음

처리:

- 보고서 카드 비활성
- 생성되지 않은 이유 문장 표시

## 10.5 네트워크 오류

문제:

- API 조회 실패

처리:

- 재시도 버튼
- 마지막 정상 데이터 유지 가능 여부 검토
- 기술 오류 문장만 보이지 않도록 일반 설명 추가

## 11. UX 원칙

- 현재 회사는 항상 상단에 보인다.
- 현재 run 기준은 항상 분명해야 한다.
- 업로드와 저장 상태는 구분한다.
- 실행 준비와 실행 완료를 구분한다.
- 긴 실행은 polling 기준으로 계속 갱신한다.
- 상태 배지 옆에 설명 문장을 같이 둔다.
- 사용자가 다음에 무엇을 할 수 있는지 액션 버튼이 분명해야 한다.

## 12. 이 문서가 다음 문서에 주는 영향

이 문서를 기준으로 다음을 상세 설계한다.

- 페이지 구조
- API 목록
- 상태관리 구조
- 컴포넌트 구조
- 에러 상태 설계
